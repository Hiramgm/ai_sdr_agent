01Overview & Rationale

Why this project, and what it automates.

The problem

Outbound sales development is slow, repetitive, and hard to do well at scale. SDRs spend hours finding leads, researching each one, writing personalized messages, chasing replies, and scheduling meetings. The work is high-volume but quality-sensitive — generic blasts get ignored, and good personalization doesn't scale with headcount.

The solution

AI SDR is an autonomous, multi-agent system. A business defines its Ideal Customer Profile (ICP); the platform then ingests and enriches matching leads, scores their fit, researches each one for context, writes a tailored first message plus a follow-up sequence, classifies and responds to replies, and books meetings — while tracking pipeline, message quality, and full observability.

Why it's strong for interviews & learning: it spans the full production AI lifecycle — ingestion + enrichment, RAG-based research, multi-agent orchestration (LangGraph), async workers, evaluation (RAGAS), and observability (Langfuse) — applied to a real, revenue-generating business process rather than a demo chatbot.

Find & Score

Ingest and enrich leads, then score each against the ICP with explanations.

Research & Write

RAG research builds context; writer + reviewer agents craft personalized outreach.

Reply & Book

Reply agent classifies intent and responds; scheduling agent books meetings.

02Core Concepts

The primitives the whole system revolves around.

RawLead → Lead

A lead as ingested from a source, then normalized + enriched (domain, seniority, region) and ready for scoring.

ICP

Ideal Customer Profile — target titles, seniorities, regions, industries used to score lead fit.

ICP Score

A fit score with human-readable reasons, used to prioritize who gets outreach.

Campaign

An outreach sequence (first message + follow-ups) targeting a set of leads.

Message & Reply

An outbound message and the prospect's response, classified by intent (interested / objection / not now / book).

Meeting

A booked slot resulting from a positive reply, tracked through the pipeline.

Communication style

  • Asynchronous (queue-based): ingestion → enrichment → scoring → persistence run as Redis/RQ jobs with retries.
  • Synchronous (request/response): dashboard ↔ API for defining ICP, launching campaigns, and reading pipeline.
  • Stateful graph: LangGraph passes a shared state (lead, research, draft, review, reply) between agent nodes.
  • Webhook (inbound): prospect replies arrive via webhook → reply agent → scheduling agent.
  • Side-channel (tracing): every node emits spans to Langfuse without affecting the main flow.

127-Day Build Plan

Each day ends with something runnable. Day 1 is implemented.

Day 1
Ingestion + enrichment + ICP scoring. Dependency-free pipeline turning raw leads into enriched, scored leads with a report. Implemented
Day 2
Databases. PostgreSQL (leads, campaigns) + MongoDB (raw data, logs) persistence.
Day 3
Vector memory + research agent. Pinecone lead memory and a RAG research agent (Groq).
Day 4
Outreach workflow. LangGraph: research → write → review → send.
Day 5
Replies + scheduling + async. Reply agent, scheduling agent, Redis async workers.
Day 6
Eval & observability. RAGAS message-quality evals + Langfuse/Phoenix tracing.
Day 7
Ship. Dockerize, deploy, dashboard, README polish, demo.
